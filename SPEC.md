# Spec: Neuron Agent Engine

## Objective

A TypeScript runtime engine for event-driven, **inverted-tree** multi-agent systems.  
Events enter at the **leaf (bottom) layer**, propagate upward through parent agents, and the **root produces the final output**.  
Each agent receives signals from its children, synthesizes/aggregates them via Claude LLM + tools, and emits one signal to its parent.  
The tree is defined by a config file (generated by an external UI project).  
**Target users:** developers building LLM pipelines; Phase 1 is a personal/internal tool.

---

## Commands

```bash
npm run build                                    # Compile TypeScript
npm run start -- --config <path.yaml>            # Start engine + API server (long-running service)
npm run start -- --config <path.yaml> --debug    # Start with debug trace logging enabled
npm run dev -- --config <path.yaml>              # ts-node watch mode
npm run dev -- --config <path.yaml> --debug      # ts-node watch mode with debug logging
npm test                                         # Run test suite
```

All signals are submitted via `POST /nodes/:id/signal` or `POST /run` — no CLI event injection.

### Runtime
- Loads tree config once at startup
- Starts HTTP API server (default port `3000`)
- Waits for signals via API — runs until killed (Ctrl+C / SIGTERM)
- No CLI event injection — all input goes through the API
- `--debug` flag (or `DEBUG=true` env var) enables trace log file output

## Config Format (YAML)

The tree is described top-down (root has `children`), but **execution flows bottom-up**.  
Leaf agents (no `children`) receive the raw event first; their output propagates up to the root.

```yaml
version: 1
root: root_agent
agents:
  - id: root_agent
    prompt: "You receive synthesized signals from your children and produce a final answer."
    model: claude-sonnet-4-6
    tools: [file, http]
    children: [agent_a, agent_b]   # standard tree structure; runs last

  - id: agent_a
    prompt: "You receive the raw event, extract type-A data, and emit a structured signal."
    model: claude-sonnet-4-6
    tools: [file]
    children: []                   # leaf: runs first, receives raw event

  - id: agent_b
    prompt: "You receive the raw event, call an external API, and emit a structured signal."
    model: claude-sonnet-4-6
    tools: [http]
    children: []                   # leaf: runs first, receives raw event
```

## Core Types

```ts
interface RawEvent {
  id: string;
  source: string;        // "system" | external identifier
  payload: unknown;      // structured JSON from external source
  timestamp: string;
}

interface Signal {
  id: string;
  fromAgent: string;
  toAgent: string;
  severity: "critical" | "warning" | "info";  // set by LLM, inherited by node status
  payload: unknown;      // structured JSON — agent's output (schema TBD per project)
  trace: TraceEntry[];   // cumulative trail from the originating raw event upward
  timestamp: string;
}

interface TraceEntry {
  agentId: string;
  summary: string;       // short human-readable summary of what this agent observed/decided
  firedAt: string;
}

interface AgentConfig {
  id: string;
  prompt: string;
  model: string;
  tools: string[];
  children: string[];    // [] = leaf (receives raw events); non-empty = receives signals from children
}

interface TreeConfig {
  version: number;
  root: string;          // id of the root (final output) agent
  agents: AgentConfig[];
}
```

## Execution Model

**Flow: leaves → … → root (bottom-up, queue-driven)**

1. Service starts, loads `--config` YAML, starts HTTP API server.
2. Engine validates config, builds agent map, starts each agent's queue consumer.
3. Raw event is pushed into the queue of **every leaf agent** (`children: []`) simultaneously.
4. Each agent runs a **FIFO queue, single consumer** — processes one signal at a time:
   a. Dequeue next signal (RawEvent or Signal from a child).
   b. Build prompt: system prompt + signal payload + trace history.
   c. Call Claude with enabled tools; Claude may invoke tools iteratively.
   d. LLM decides: **fire** (emit signal to parent) or **stay silent** (drop, nothing emitted).
   e. If firing: construct a new `Signal` — append a `TraceEntry` (agentId + short summary) to the incoming trace, set payload to LLM output, send to parent's queue.
5. Parent agents receive signals from children into their own queues and repeat step 4.
6. Root agent's output (if it fires) is printed to stdout.
7. Every processed signal writes a trace log entry: `{ agentId, inputSignal, fired, toolCalls, output, durationMs }`.

## Tools — Phase 1

### File I/O (`file`)
- `file_read(path: string) → string`
- `file_write(path: string, content: string) → void`
- Paths resolved relative to a configurable `workdir` (default: `./workspace`).

### HTTP (`http`)
- `http_get(url: string, headers?: Record<string, string>) → string`
- `http_post(url: string, body: string, headers?: Record<string, string>) → string`
- Response body returned as string (JSON or text).

### Context (`context`)
- `context_get(key: string) → string | null`
- `context_set(key: string, value: string) → void`
- `context_delete(key: string) → void`
- Each node has its own isolated `Map<string, string>` — LLM decides what to store and retrieve.
- In-memory only; resets on engine restart.

## Code Style

- TypeScript strict mode (`"strict": true`)
- **LangGraph JS** (`@langchain/langgraph`) for graph execution, state, and node routing
- **Anthropic SDK** via `@langchain/anthropic` for LLM calls
- **Hono** with `@hono/zod-openapi` for typed routes + auto-generated OpenAPI schema
- **Swagger UI** served at `/doc` via `@hono/swagger-ui`
- Zod for config validation at load time
- Plain functions for tools and node handlers — no unnecessary classes
- No `any` — use `unknown` and narrow explicitly
- Errors bubble up with context: `throw new Error(\`[agent:${id}] tool failed: ${e.message}\`)`

## Observability

### Console (stdout)

One line per event, always visible:

```
→ [agent_id] event received   <source> | <truncated payload summary>
✓ [agent_id] fire             severity=warning  "<summary>"
✗ [agent_id] silent
! [agent_id] error            <message>
```

- `→` on incoming signal/event (leaf nodes show the raw event; parent nodes show the child signal)
- `✓` when the agent fires, with severity and one-sentence summary
- `✗` when the agent stays silent
- `!` on error

No raw JSON objects on stdout. Stderr: engine startup errors only.

### Trace log (debug mode only)

Written to `./logs/trace-<runId>.jsonl` **only when `--debug` flag or `DEBUG=true` env var is set**.  
Each line is a JSON object: `{ agentId, fired, input, output, timestamp }`.  
In normal mode no log files are created.

## Status API

A lightweight HTTP server (Hono or Express) runs alongside the engine and exposes node status for polling.

### Endpoints

```
GET  /doc                          → Swagger UI (interactive API explorer)
GET  /openapi.json                 → OpenAPI 3.0 schema
GET  /config                       → loaded tree config (TreeConfig as JSON, for UI visualization)
GET  /nodes                        → all nodes + current status
GET  /nodes/:id                    → single node status + last signal
POST /nodes/:id/signal             → push a signal directly into a specific node's queue
GET  /nodes/:id/context            → read all context keys for a node
DELETE /nodes/:id/context          → clear all context for a node
POST /run                          → submit a raw event to all leaf nodes, returns runId
GET  /runs/:runId                  → status of a specific run (pending | running | complete | error)
```

### POST /nodes/:id/signal

Request body:
```json
{
  "payload": { },        // any JSON
  "source": "system",    // who is sending
  "trace": []            // optional — carry existing trace or start fresh
}
```

Response:
```json
{ "queued": true, "nodeId": "extractor_a", "queueDepth": 1 }
```

This lets you target any node directly — bypass the full tree, inject mid-tree, or test a single agent in isolation.

### Node Status Shape

```ts
type NodeState = "idle" | "processing" | "silent" | "error";
type AlertSeverity = "critical" | "warning" | "info";

interface NodeStatus {
  id: string;
  state: NodeState;          // current activity state
  severity: AlertSeverity | null;  // severity of last fired signal, null if silent/idle
  lastSignalAt: string | null;
  lastSignal: Signal | null;
  errorMessage: string | null;
  processedCount: number;
}
```

A node that fired inherits its `severity` from the signal payload. A node that is `silent` or `idle` has `severity: null`.

### Behaviour
- Status is **in-memory**, resets on engine restart
- Each run gets a `runId` (UUID); multiple runs can be in flight simultaneously
- `GET /nodes/:id` always reflects the node's **current** status regardless of run

## Boundaries

| Always | Ask First | Never |
|--------|-----------|-------|
| Validate config on load, fail fast | Before overwriting an existing file | Store API keys in config YAML |
| Log trace per agent | Before making POST requests to unknown hosts | Execute arbitrary code (no `exec` tool in Phase 1) |
| Handle tool errors gracefully (catch + surface in signal) | | Mutate the input event — always create new Signal objects |

- [ ] Raw event pushed to all leaf queues simultaneously
- [ ] Each agent processes queue FIFO, one at a time, fires or stays silent
- [ ] Fired signals carry cumulative trace from origin to root
- [ ] Root output printed to stdout

- [ ] File I/O and HTTP tools work end-to-end
